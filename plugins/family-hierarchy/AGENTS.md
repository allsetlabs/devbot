# Family Hierarchy Plugin

An interactive, editable **family-tree graph** for a 1978 Tamil genealogy (the descendants of
*Aayngudi Vellaiyan Chettiar* of Kanadukathan & Pallathur). It's a **DevBot plugin** — reached at
`/plugins/family-hierarchy` (via the **Plugins** hub; the shared app `Header` shows
"Plugins › Family Hierarchy"). The legacy `/family-hierarchy` path redirects to the canonical one.

## The one hard rule: NO database

Unlike baby-logs / lawn-care, this plugin has **no SQLite, no Drizzle**. The entire tree is a
single JSON file at **`modules/memory/family_hierarchy.json`** (the repo's memory module). The
backend exposes exactly two endpoints (mounted at `/api/plugins/family-hierarchy`):

- `GET /tree` — read the file
- `PUT /tree` — overwrite the file with the whole tree (the app sends the entire JSON on every edit)

Path is `FAMILY_HIERARCHY_PATH` env or resolved relative to the router. If you add fields, keep
`PUT` writing the **whole** document — there are no partial updates.

## Structure

```
backend/routes.ts     # getFamilyHierarchyRouter(): GET/PUT /tree, fs-backed (no DB)
frontend/
  types.ts            # Person, FamilyTree
  api.ts              # getFamilyTree / putFamilyTree (hits the DevBot backend + X-API-Key)
  lib/tree.ts         # pure helpers: visible set, lineage, expand/collapse, immutable edits, merge-time ids
  components/         # PersonNode, LeftPanel, GraphToolbar, EditPersonModal
  pages/FamilyHierarchy.tsx   # the page (state + orchestration); renders no top bar — the app Header does
```

The interactive graph itself is a **forge** component (`@allsetlabs/forge/components/ui/hierarchy-graph`)
— a generic pan/zoom top-down tree renderer. Node visuals + all family logic live here in the plugin.

## API

`frontend/api.ts` calls the DevBot backend on its own port with the shared `X-API-Key`
(`VITE_BACKEND_PORT` / `VITE_API_KEY` from the app `.env`) — same convention as every other DevBot
plugin. `GET/PUT /api/plugins/family-hierarchy/tree` read/write the memory JSON file (no DB).

## Dark theme + forge gotcha

The page is deliberately **dark** (glowing-node design). It renders inside a `dark` wrapper, and
modal/tooltip portals (which mount at `<body>`) also get a `dark` class so they match. Forge
utilities only apply inside `#forge-app-root`, which the DevBot app sets on `<body>` (`applyToBody`).

## Where the data came from

The JSON is generated by merging 34 per-page OCR passes. The pipeline + scripts live in
`.tmp/ancestor-hierarchy/reconciled/` (`merge_family.py` produces `family_hierarchy.json`; it
canonicalises recurring OCR spelling variants so the shared root→5-sons→… spine dedupes into single
people — 832 raw nodes → ~411 unique). Re-run it to regenerate the file from source.

## Run it

It's part of DevBot — start DevBot (`make start` from the repo root; a restart is needed to pick up
the backend router) and open **Plugins → Family Hierarchy**.
